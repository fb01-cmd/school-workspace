## 📌 1. 전체 개발 로드맵 (Phase 1 ~ Phase 8)

### Phase 1. 학적 관리 핵심 기능 (완료 ✅)
* **Google OAuth + RBAC**: 이메일 정규식(`/^\d{5}@hmh\.or\.kr$/`)을 이용한 학생(`student`), 미승인 교사(`teacher`), 수퍼어드민(`super_admin`) 자동 분류 및 라우팅 가드.
* **GWS 조직단위(OU) 및 그룹스(Groups) 관리**: 계층 트리 및 반별 메일링 그룹 생성/삭제 자동화.
* **감사 로그(Audit Log)**: 시스템 내 모든 계정 생성, 수정, 삭제 이력 Firestore 적재.

### Phase 2. 교직원 계정 생애주기 관리 (완료 ✅)
* **교사 전입**: 계정 일괄/단건 생성 시 **4대 교사 연동 그룹**(`ts@`, `classroom_teachers@`, `hmhteacher@`, `hmh_teachers@`) 가입 처리 및 Firebase Auth UID 청소.
* **교사 전출**: 4대 그룹 즉시 강제 탈퇴 처리 ➔ 메일/구글챗 안내 ➔ 교사가 직접 백업 데드라인 선택(최대 1년) ➔ 데드라인 경과 시 자동 일시정지 및 30일 후 자동 삭제(크론 배치).
* **명예퇴임**: 계정을 삭제하지 않고 **'OB 보존실' OU**로 이동 및 그룹만 탈퇴하여 이메일 영구 보존.

### Phase 3. 졸업 동의서 터치 서명 및 보관 강화 (완료 ✅)
* **터치 서명 패드**: 학생 포털 내 HTML5 Canvas 기반 친필 서명 구현.
* **영구 보존 컬렉션 (`graduation_consents`)**: 유저 계정이 삭제되더라도 행정 증빙을 위해 서명 데이터를 2~3년간 별도로 안전하게 보존.
* **인쇄 지원**: 감사 제출용 A4 인쇄 CSS 미디어 쿼리 최적화.

### Phase 3.5. UX 성능 개선 및 삭제 계정 복원 (완료 ✅) — 2026-07-16
* **클라이언트 메모리 캐시 (`clientCache.ts`)**: 브라우저 탭 단위의 인메모리 캐시(TTL 5분) 모듈을 구축하여 메뉴 전환 시 로딩 스피너 없이 즉시 렌더링.

* **AutocompleteInput 로컬 검색 전환**: 전출 교사 검색 등 사용자 자동완성을 서버 API 디바운스 방식(300ms)에서 캐시 로컬 필터링(50ms)으로 전환하여 체감 반응속도 대폭 향상.

### Phase 4. 구글 클래스룸 로스터 강제 배정 (완료 ✅) — 2026-07-16
* **배경**: 유료 기능인 `Class Tools`는 학생이 클래스룸에 단순 초대가 아니라 **로스터에 수락 상태로 등록**되어야만 정상 작동함. 기존에는 `Classright` 시트 부가기능으로 수퍼어드민이 수동 처리하던 영역을 웹 앱 내로 내재화.
* **동작 프로세스 및 구현 상세**:
  1. **최고 관리자 계정 사칭(Impersonation)**: 일반 교사는 Google API 규정상 학생을 클래스룸에 직접 추가할 권한이 없어 `403 Unauthorized` 에러가 발생함. 이를 우회하기 위해 학생 추가/제거 API 실행 시 백엔드에서 최고 관리자(`GOOGLE_WORKSPACE_ADMIN_EMAIL`) 권한으로 사칭하여 강제 배정을 즉시 수행하도록 설계.
  2. **동적 학급 스캔**: 학급 수(반 개수)를 하드코딩하지 않고, 클라이언트 캐시(`users:all`)를 통해 선택 학년의 조직단위(OU) 내 학생 목록을 조회함. 학생들의 학번 패턴(예: `10203` -> 2반)을 실시간으로 파싱하여 고유하게 존재하는 반들만 드롭다운에 동적 렌더링되도록 구현. 내년 학급 변동 시에도 완벽하게 유연 대응 가능.
  3. **초대 수락 과정 생략**: 학생들이 구글 클래스룸 초대 메일을 직접 열고 수락할 때까지 대기할 필요 없이, 즉시 소속 멤버 목록에 "가입됨" 상태로 강제 안착시킴.
  4. **단어 및 UI 개선**: 학생들을 대할 때 어색하고 불필요한 은유인 '바구니'라는 단어를 '배정 대기 명단' 등으로 전면 수정하고, 사이드바를 포함한 메인 명칭을 '클래스룸 학생 강제 배정'으로 격상.
  5. **예외 처리 한글화**: 존재하지 않는 계정(404)이나 이미 가입된 학생(409) 등 구글 API가 반환하는 원시 에러들을 한글 예외 처리하여 사용자(교사) 피드백 품질 개선.

### Phase 5. 크롬 관리 북마크 즉시 배정 및 실시간 히스토리 로그 (완료 ✅) — 2026-07-16

* **구현 핵심**:
  1. **맞춤형 북마크 트리 편집기 (`BookmarkTreeEditor.tsx`)**: 학교 웹 플랫폼 안에서 폴더 생성, 북마크 노드 추가/삭제 및 Drag & Drop 편집이 가능한 간편 UI 제공. 기본 닫힘 폴더 구조, 드래그 중 자동 펼치기(650ms 호버), 모달 기반 추가/수정, 삭제 재확인 등 UX 세부 구현 완료.

  4. **북마크 변경 이력 로그 (스마트 diff 뷰)**: 누가 언제 어떤 북마크를 변경(추가/삭제/순서변경)했는지 구조적으로 비교하는 diff 엔진(`computeBookmarkDiff`) 구현. 전체 트리 덤프 대신 추가/삭제/이동 항목만 표시.
  5. **AuthContext 프리페치 연동**: OU 목록을 페이지 진입 시 별도 API 호출 없이 `useAuth().orgUnits`에서 즉시 사용하도록 최적화.

### Phase 5.8. 학기말 클래스룸·캘린더·드라이브 일괄 정리 (배포 승인 ✅ — 후속 과제 4건 남음) — 2026-07-24 스펙 작성

> **배경**: 매년 학기말 보관처리를 안 하는 교사가 많아 활성 클래스룸이 누적되며 로딩 지연·검색 저하가 발생. 학생은 설정에 따라 클래스룸을 스스로 나갈 수 없어 교사의 보관처리에 전적으로 의존. 보관처리해도 (1) 클래스룸 캘린더 구독이 자동 해지되지 않고 (2) 드라이브 폴더가 자동으로 "이전년도" 폴더로 옮겨지지 않아 정리가 반쪽으로 끝남. Claude가 설계 담당 — 계정/데이터가 아닌 도메인 전체 상태(캘린더·드라이브)를 대량으로 건드리는 작업이라 되돌릴 수 없는 작업에 준하는 판단이 필요함.

**사전 조사 결과 (2026-07-24, Explore 서브에이전트)**:
- `workspace.ts`에 Classroom 코스 생성/로스터 함수는 있으나 **보관처리(courses.patch, courseState=ARCHIVED) 함수는 없음** — 신규 구현 필요.
- **Calendar API, Drive API는 저장소 전체에 코드/스코프가 전혀 없음** — 서비스 계정 도메인 위임(domain-wide delegation)에 `calendar`(캘린더리스트 삭제엔 read-only 스코프로 부족, 전체 `calendar` 스코프 필요) 및 `drive`(폴더 이동은 `drive.file`로는 불충분 — 앱이 생성하지 않은 기존 폴더라 광범위 `drive` 스코프 필요) 스코프를 Admin Console에서 **먼저 추가해야 착수 가능**. 이는 배포 체크리스트의 "Firebase 인증 관리자" 역할 부여와 동일한 성격의 사전 조건이므로 `deployment_checklist.md`에도 반영 필요.
- 기존 lifecycle UI 패턴(`StudentLifecycle.tsx`/`TeacherLifecycle.tsx`)은 `window.confirm()` + `Promise.allSettled` 결과 표시 방식으로, **미리보기 후 개별 검토·수정하는 위저드는 아직 이 프로젝트에 없음.** 이번 기능은 사용자가 직접 "확인하면서 해야 한다"고 명시했으므로, 기존 패턴을 그대로 따르지 않고 **미리보기·개별 수정 단계를 신설**하는 것으로 판단(§아래 UX 설계 참조).

**핵심 설계 결정 6가지**:

1. **연도 접두어 오탐지 방지 — 항상 사람이 최종 확인한다.**
   `^20\d{2}[\s\-\[]` 류 정규식으로 "이미 접두어 있음/없음"을 1차 자동 분류하지만, 분류 결과를 실행 전에 **미리보기 표에서 교사별·클래스룸별로 노출**하고 관리자가 개별 오탐을 직접 수정할 수 있게 한다. 자동판별을 신뢰하고 바로 실행하지 않는다 — 이름 오류는 학생·학부모에게 그대로 노출되는 영역이라 계정 삭제보다 오히려 사용자 체감 리스크가 큼.

2. **작업당 4단계 파이프라인 + 단계별 상태 추적, idempotent 재실행.**
   클래스룸 1개당 `이름변경 → 보관처리 → 캘린더 구독취소 → 드라이브 폴더 이동` 순서를 지키고, 각 단계를 개별 성공/실패로 기록한다. 재실행 시 이미 완료된 단계는 **매번 라이브 API 상태를 조회해 건너뛴다** (Firestore 플래그를 신뢰 소스로 삼지 않음 — 중간에 교사가 수동으로 손댔을 가능성을 감안한 자기치유형 설계). Firestore(`classroom_cleanup_logs` 신설 컬렉션)에는 감사 로그 및 **원복에 필요한 이전 상태(원래 이름, 원래 드라이브 부모 폴더 ID, 캘린더 ID)** 를 반드시 남긴다.

3. **모든 단계는 원복 가능하게 설계한다 (계정 삭제류와 다름).**
   이름변경·보관해제(unarchive)·캘린더 재구독·드라이브 폴더 원위치 복귀 모두 기술적으로 가능 — Phase 3.5의 "삭제된 사용자 복원" 탭과 동일한 철학으로 이 화면에도 **"최근 정리 내역 되돌리기"** 를 함께 제공한다. 구글 관리 콘솔에서는 보관된 클래스룸을 찾아 해제하기가 번거로우므로 이 기능 자체가 부가가치가 있음.

4. **실행 주체는 교사 본인 — 관리자는 실행 권한 없이 현황 조회만 한다. (2026-07-24 수정: 최초 설계였던 "슈퍼어드민 일괄 실행"을 폐기)**
   근거: 관리자가 일괄 실행하면 오탐 이름변경·오보관 등 실수 시 책임이 관리자에게 집중되어 민원의 표적이 됨. 교사가 스스로 안 하는 진짜 이유는 "귀찮음"이지 "못 해서"가 아니므로, 번거로움 자체를 자동화로 없애면 직접 할 유인이 생긴다는 사용자 판단을 반영. 백엔드는 **로그인한 교사 본인 이메일로 사칭(impersonation)** 하여 API를 호출한다 — 도메인 관리자 사칭이 아니라 본인 소유 리소스에 대한 본인 사칭이므로 권한 범위가 가장 좁고, 관리자 사칭보다 성공 가능성도 높을 것으로 예상(§검증 필요 항목 참조). 관리자 화면은 "누가 몇 개 남았는지"만 보여주는 읽기전용 현황 대시보드로 한정하고, 관리자가 남 대신 실행하는 버튼은 두지 않는다.

5. **매년 2월부터 로그인 시 지속 알림 — 단, 스누즈·예외처리로 피로도를 관리한다.**
   - **트리거**: 매년 2월 1일부터, 본인 소유 활성(미보관) 클래스룸 중 "정리 대상"(연도 접두어 없음 등)이 하나라도 있으면 알림 대상이 됨.
   - **에스컬레이션 단계**: 로그인 시 닫을 수 있는 배너(1~2주차) → 닫아도 다음 로그인마다 재노출(3주차~) → 이메일/구글챗 1회 발송(개학 D-14 시점, 기존 `hmnotice@` 발신자 규칙 재사용) → 2월 말 신학기 임박 시 대시보드 진입 즉시 모달로 격상.
   - **피로도 관리 장치(필수)**: 클래스룸별로 "정리 대상 제외" 표시(동아리반 등 연중 유지 클래스룸 대응) 및 "일주일 후 다시 알림" 스누즈를 반드시 제공한다. 이게 없으면 알림 자체가 새로운 민원 대상이 되어, 관리자 일괄 실행을 폐기하며 없앤 리스크를 알림 설계가 그대로 재도입하게 된다.
   - **구현**: 기존 `lifecycle/cron` 크론 인프라를 재사용해 이메일 발송 등 시점 기반 에스컬레이션만 스케줄링한다. 배너에 표시할 "남은 정리 대상 개수"는 아래 결정 #6의 판별식이 `courses.list`의 `creationTime`/`courseState`만으로 계산 가능해 **매 로그인마다 라이브로 재계산해도 비용 문제가 없다** — 별도 캐싱 인프라 불필요.

6. **"정리 대상" 판별은 생성일자(`creationTime`) 기반으로 하되, 학사력의 실제 운영 관행을 반영한 학년도 매핑 규칙을 쓴다. (2026-07-24, 사용자 확인)**
   - Classroom Course 리소스에는 `creationTime` 필드가 존재한다(§검증 항목에서 실제 응답값 확인 필요 — 지금까지 조사된 코드베이스에는 이 필드를 다뤄본 이력이 없음).
   - **사용자 확인 사항 (1차)**: 교사는 이미 방학에 들어간 뒤에 "곧 끝나는" 학년도용 클래스룸을 새로 만들지 않는다 — 생성월만으로 충분히 구분 가능.
   - **사용자 확인 사항 (2차 수정, 2026-07-24)**: 단 **1월은 예외에 넣으면 안 된다.** 겨울방학 보충수업용 클래스룸을 1월에 새로 만드는 경우가 있는데, 이건 **아직 끝나지 않은(진행 중인) 학년도**에 속하는 정상적인 활동이다. 만약 1월 생성분도 "다음 학년도"로 당겨버리면, 학년도가 실제로 바뀐 뒤에도 이 보충수업 코스가 "현재 학년도 소속"으로 오분류되어 **한 해 내내 정리 대상에서 누락**되는 역효과가 생긴다(놓치는 게 없느니만 못한 결과 — 클러터 감소라는 이 기능의 목적을 정면으로 해침). **2월만** 예외로 남긴다 — 2월엔 아직 진행 중인 보충수업이 실제로 드물고, 신학기 준비로 미리 만드는 경우가 대부분이라는 판단.
   - **학년도 매핑 규칙**: `creationTime`의 (연, 월)을 `(Y, M)`이라 할 때,
     - `M`이 3~12월이면 → 그 코스는 **`Y`학년도**(해당 연도 3월 시작) 소속.
     - `M`이 1월이면 → 그 코스는 **`Y-1`학년도**(직전 연도 3월 시작, 즉 아직 진행 중인 학년도) 소속 — 표준 학사력 매핑 그대로, 예외 없음.
     - `M`이 2월이면 → 그 코스는 **`Y`학년도**(같은 연도 3월에 시작하는, 즉 다음 학년도) 소속으로 당겨서 간주 — 신학기 준비용 사전 생성 클래스룸을 여기서 거른다.
   - **트레이드오프 인지**: 아주 드물게 1월에 다음 학년도용 클래스룸을 미리 만드는 교사가 있다면, 그 코스는 학년도가 바뀔 때까지 "과거 학년도"로 오분류되어 정리 대상 알림이 한 번 잘못 뜰 수 있다. 이건 결정 #5의 "정리 대상 제외" 스누즈로 즉시 해소 가능한 **약한 오탐**이라, 앞서 설명한 "1년 내내 놓치는" **강한 미탐**보다 훨씬 안전하다고 판단해 감수한다.
   - **정리 대상 조건**: 오늘 날짜 기준 "현재 학년도"보다 **과거 학년도**에 매핑되면서 `courseState = ACTIVE`(아직 미보관)인 코스만 정리 대상으로 플래그한다.
   - 이 규칙은 `courses.list` 응답의 `creationTime`, `courseState` 두 필드만으로 계산되므로 로스터·코스워크 조회가 필요 없다 — 결정 #5의 라이브 재계산 부담도 사실상 사라진다.
